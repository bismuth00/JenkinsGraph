# JenkinsGraph

Jenkins のビルド履歴を定期収集して SQLite に蓄積し、静的 HTML レポート
(失敗期間のタイムライン + 日別失敗率ヒートマップ) を生成するツール。

```
cron ──> collect.py ──> jenkins.db (SQLite) ──> report.py ──> report.html
          (差分取得)                              (集計 + ECharts 埋め込み)
```

Jenkins はビルド履歴をローテーションで破棄するため、履歴が消える前に
自前 DB へ写し続けるのがこの構成の肝。集計はレポート生成時に行うので、
表示要件が変わっても再収集は不要。

## セットアップ

Python 3.11 以上 (tomllib を使用)。

```sh
python -m venv .venv
.venv/bin/pip install -r requirements.txt   # Windows: .venv\Scripts\pip
cp config.example.toml config.toml
# config.toml に Jenkins の URL / ユーザー / API トークンを記入
```

API トークンは Jenkins の「ユーザー設定 > API トークン」で発行した
読み取り専用ユーザーのものを推奨。`config.toml` に書きたくない場合は
環境変数 `JENKINS_TOKEN` でも渡せる。

## 使い方

```sh
python collect.py   # Jenkins からビルド履歴を差分取得して DB に蓄積
python report.py    # DB を集計して report.html を生成
```

`report.html` は単一ファイルなので、ブラウザで直接開くか、
ファイルサーバー・S3 などに置いて共有する。

### cron 設定例 (1 時間ごと)

```cron
0 * * * * cd /path/to/JenkinsGraph && .venv/bin/python collect.py && .venv/bin/python report.py >> collect.log 2>&1
```

Windows ならタスクスケジューラで同等のコマンドを登録する。

## レポートの見方

レポートは 4 つのタブに分かれている。

- **概要タブ**: 期間を指定して全体の傾向を見る (以下の 3 チャート)
- **ノードタブ**: 1 日単位で「どのノードで何が実行されたか」を見る。
  行 = ノードで、ビルド実行が結果の濃色の帯 (並列実行は行内で縦に積む)、
  オフライン期間がグレーの帯 (collect.py のサンプリング由来なので粒度は
  収集間隔)。ノード名の下に稼働率 (実行時間 ÷ 経過時間×エグゼキュータ数)・
  ビルド回数・失敗数が出る。帯クリックでビルドページ、ノード名クリックで
  ノードのページが開く。実行ノード (builtOn) が取れないビルド (主に
  Pipeline) はこのタブには表示されない。Jenkins から削除されたノード
  (最新のサンプリングに現れないノード) はグレーの「(削除済み)」表示になり、
  その日に表示するビルドやオフライン帯がなければ行ごと出ない
  (ノードページへのクリックも無効)。ジョブの絞り込みの対象外で、
  常に全ジョブのビルドを表示する (絞り込みバーもこのタブでは非表示)。
  代わりにノード専用の絞り込み (名前の部分一致検索) と並び順
  (名前順 / 稼働率順 / ビルド回数順) がある。
- **パイプラインタブ**: 1 日単位で「他のジョブを呼び出したビルド」ごとに
  プロファイル型のチャートを表示する。行 = ジョブ (先頭が呼び出し元)、
  バー = 各ビルドの実行 (結果の濃色) とキュー待ち (黄色) で、パイプラインの
  処理の中でどのタイミングで下流ジョブが開始・終了したかが分かる。
  同じジョブの並列実行は行内でサブレーンに縦積みされ、重ならない。
  呼び出し関係はビルドの UpstreamCause から取り、孫以降の呼び出しも
  同じチャートにまとめる。バークリックでビルドページ、ジョブ名クリックで
  ジョブページが開く。検索でプロファイルを絞り込める (ジョブ名の部分一致、
  または「#66」形式のビルド番号一致。呼び出し先が一致した場合も
  それを含むプロファイルが表示される)。並び順は開始時刻順 /
  全体が長い順 (キュー投入〜最後の実行終了) / 待機の合計が長い順。
  各プロファイルのヘッダに全体の長さと待機合計も表示される。
- **詳細タイムラインタブ**: 1 日単位でキュー待ち・ビルド実行・待機を見る。
  カレンダーと前日/翌日ボタンで日付を移動する。キュー待ち (スケジュール〜
  実行開始) は明るい黄色、ビルド実行中は結果の色の濃い版 (成功=濃い緑、
  失敗=濃い赤) で太く表示され、ビルド後〜次のビルドまでは直前の結果の
  通常色で表示される。ジョブ名の下にはその日の時間内訳 (成功/失敗などの
  状態・ビルド実行・キュー待機の割合) とビルド回数・結果の内訳が出る。

ジョブの絞り込み (検索 / 失敗のみ / 並び順) はタブの上にあり、
概要タブと詳細タイムラインタブに共通で効く (ノードタブでは非表示・無効)。
並び順の失敗率・悪化の判定は概要タブの表示期間で計算される。

- **表示期間**: ページ上部のカレンダーで開始日・終了日を自由に指定できる
  (集計はブラウザ側で行うため再生成は不要)。「直近 7日/14日/...」ボタンは
  日付を埋めるショートカット。選択できる範囲はレポートに埋め込んだ
  `report.days` 日分まで。
- **表示ジョブの絞り込み**: ジョブ名の部分一致検索 (スペース区切りで OR、
  「app/」のようにフォルダ名でも絞れる)、「失敗があったジョブのみ」トグル、
  並び順の切り替え (名前順 / 失敗率ワースト順 / 悪化順)。全チャートに反映され、
  チャートの高さも表示ジョブ数に合わせて縮む。
  Jenkins 上で無効化されているジョブはデフォルト非表示で、
  「無効なジョブも表示」を ON にすると表示される (有効/無効の状態は
  collect.py が実行のたびに jobs テーブルへ記録する。このテーブルが
  ない古い DB では全ジョブが有効扱い)。無効なジョブは状態が続いている
  とは言えないため、タイムラインの帯は最後のビルドの実行終了で打ち切られ、
  それ以降は空白になる (時間ベースの失敗率の分母からも除外)。
- **タイムライン**: ジョブごとに 1 行。同じ結果が続いた期間が 1 本の帯になる。
  帯をクリックすると該当ビルドの Jenkins ページが新しいタブで開く
  (概要はその区間の先頭ビルド、詳細タイムラインは各区間のビルド)。
  タイムライン・ヒートマップのジョブ名クリックでジョブのトップページが開く
  (推移グラフのジョブ名は凡例のため、クリックは表示 ON/OFF のまま)。
  失敗の帯は「最初に失敗したビルド〜次に結果が変わったビルド」の区間で、
  失敗したまま次のビルドがないジョブは現在時刻まで赤く伸びる。
  ジョブ名の下に表示期間内の成功/失敗のパーセント・ビルド数と、
  期間の前半→後半で失敗率がどう変化したかの傾向バッジ
  (↗悪化 / ↘改善 / →横ばい、±5pt が閾値) が出る。
- **失敗率の推移**: ジョブごとの 7 日移動平均 (太線) と日次の生値 (薄い細線)。
  移動平均は「直近 7 日間の失敗時間 ÷ 状態が分かっている時間」なので、
  ビルドが少ない日のノイズに引っ張られずに悪化/改善の傾向が読める。
  凡例クリックで表示の ON/OFF を切り替えられる (期間を変えても維持される)。
- **ヒートマップ**: ジョブ × 日のマス目。色は当日のビルドの失敗率
  (失敗ビルド数 ÷ ビルド数、%)、セル内の数値は当日のビルド回数。
  ビルドがなかった日は空白。

ビルド番号が連続していない区間 (Jenkins のログローテーションで間のビルドが
取得できなかった期間。lastFailedBuild などは削除されず残るため生じる) は
「状態不明」として扱い、直前のビルドの結果はその実行終了までしか描かない。
状態不明の期間はタイムラインでは空白になり、時間ベースの失敗率の分母からも
除外される (ずっと失敗していたかのような見た目になるのを防ぐ)。

推移グラフの日別失敗率は「その日のうち失敗状態だった時間の割合」。
タイムラインの帯と同じく、失敗ビルドから次に結果が変わるビルドまでを
失敗期間とみなして 1 日 (24 時間) との重なりで計算する。当日など 1 日が
終わっていない分や、ジョブの最初のビルドより前の時間は「状態不明」として
分母から除く (状態不明しかない日は線が途切れる)。傾向バッジと悪化順の
並び替えもこの時間ベースの失敗率で計算する。

## 補足・チューニング

- ジョブの絞り込みは正規表現 (`include` / `exclude`、fullName に対する
  部分一致。`include` が空なら全ジョブ、`exclude` が優先) で、
  収集用と表示用を別に設定できる:
  - `[jobs]` — **収集対象** (collect.py)。除外したジョブは収集されない
    (Jenkins 側の履歴が消えると遡って取得できない) ので絞りすぎに注意
  - `[report.timeline]` — 概要タブ (タイムライン・失敗率の推移・ヒートマップ)
    と詳細タイムラインの表示対象
  - `[report.node]` — ノードタブに表示するビルドの対象
  - `[report.pipeline]` — パイプラインタブの起点ビルドの対象
    (旧 `[report.charts]` は node / pipeline が未指定のときの
    フォールバックとして引き続き使える)
- 「失敗」の定義は `report.py` の `FAIL_RESULTS` (既定: FAILURE と UNSTABLE)。
  UNSTABLE を成功扱いにしたい場合はここを変更する。
- レポートに含める最大期間は `config.toml` の `report.days` (既定 60 日)。
  埋め込むデータ量もこれに比例するので、長くしすぎると HTML が重くなる。
- レポートのデータは差分エンコード + gzip 圧縮で埋め込まれる
  (無圧縮比で約 1/4)。展開に DecompressionStream を使うため、表示には
  モダンブラウザ (Chrome 80+ / Edge / Firefox 113+ / Safari 16.4+) が必要。
  時刻・所要時間は秒精度に丸められる。
- 概要タブの「日別失敗率の推移」と「日別ヒートマップ」は `config.toml` の
  `report.show_trend` / `report.show_heatmap` (既定 true) で非表示にできる
  (タイムラインは常に表示)。
- 差分取得に使う Jenkins API の `builds` フィールドは直近 100 件しか
  返さない。実行間隔の間に 1 ジョブが 100 回以上ビルドされる環境では
  実行頻度を上げること (初回のみ `allBuilds` で全履歴を取得する)。
- フォルダ / マルチブランチパイプラインは再帰的にたどる
  (`collect.py` の `CONTAINER_CLASS_KEYWORDS` で判定)。
- パイプラインタブの呼び出し関係は、各ビルドの UpstreamCause
  (upstreamProject / upstreamBuild) から取得する。Pipeline の build ステップや
  下流ジョブトリガーで起動されたビルドが対象で、手動起動やスケジュール起動の
  ビルドには付かない。上流ジョブがレポートの収集対象外だと関係も表示されない。
  次回の collect.py 実行以降のビルドから有効。
- ノード情報: ビルドの実行ノードは builtOn から取得する (Freestyle のみ。
  Pipeline は複数ノードで実行され得るため取得できず「ノード不明」になる)。
  ノードのオンライン/オフラインは collect.py が実行のたびに
  /computer/api/json をサンプリングして node_status テーブルに蓄積する。
  どちらも次回の collect.py 実行以降のデータから有効。
- キュー待ち時間は Jenkins の Metrics プラグインがビルドに付与する
  `queuingDurationMillis` から取得する。プラグインがない環境では 0 として
  記録され、詳細タイムラインにキュー待ちの帯が出ないだけで他は動作する。
  既存の DB は次回の collect.py 実行時に自動でカラムが追加される
  (追加前に収集済みのビルドのキュー待ちは 0 のまま)。
- `template.html` は ECharts を CDN から読み込むため、レポート閲覧時に
  インターネット接続が必要。オフライン環境では echarts.min.js を
  ローカルに置いて `<script src>` を書き換える。
